Maps each hand-written recursive-descent parser function in `src/parser/grammar/` to the official OMG KEBNF grammar rule it implements.

== Sources

* `docs/grammar/SysML-textual-bnf.kebnf` and `docs/grammar/KerML-textual-bnf.kebnf` are unmodified (except for the `tag::`/`end::` markers used below) copies of the official grammar, fetched from https://github.com/Systems-Modeling/SysML-v2-Release[Systems-Modeling/SysML-v2-Release] via the https://crates.io/crates/kebnf[`kebnf`] CLI (`kebnf --fetch-spec`).
* Every code block below is pulled live from the current source via AsciiDoc's tagged-region `include` mechanism -- it is never a stale copy-paste. If a function or grammar rule is renamed, the corresponding `tag::`/`end::` pair must move with it or the include will fail at render time.
* This document replaces the old `/// Per pest: ...` doc-comment citations that used to reference a since-deleted, hand-rolled `.pest` grammar (see `docs/grammar-gaps.adoc` for the broader parser-completeness audit this superseded work grew out of).

== How to read an entry

Each entry shows, in order: the Rust parsing function, the KEBNF production(s) it implements, and a fidelity note. "Good/full match" means the Rust structure tracks the official grammar closely. "Partial" means the Rust function is broader, narrower, or merges multiple official rules -- not necessarily a bug (a hand-written recursive-descent parser routinely collapses several small grammar rules into one function for pragmatic reasons), but worth knowing when comparing behavior against the spec. "Does not correspond well" flags custom/legacy logic with no real official counterpart.

NOTE: A few function doc comments in the KerML expression module were found, during this mapping work, to describe a *different, nearby* function than the one they're attached to (a pre-existing misattachment, not introduced by this pass) -- see the notes on `parse_ternary_conditional`, `parse_literal`, `looks_like_typed_parameter`, and `handle_feature_chain` below.
